# Rules: Ports Layer

## Context & Responsibility
Abstract interfaces (Protocols) that define contracts between application and infrastructure. This layer contains no implementation—only type definitions that concrete adapters must fulfill.

## Architectural Contract
- All ports use Python `Protocol` (structural typing)
- No ABC (Abstract Base Class) - protocols are more flexible
- All port methods are async
- No implementation code whatsoever
- Ports live in `ports/*.py` (flat structure)

## Quality Standards
- Complete type signatures on all methods (no `Any`, no `...`)
- Docstrings on all protocols describing behavior
- Return types must be specific (not generic `object`)
- All async methods

## Naming Conventions
- Port name ends with `Port` (e.g., `EmbeddingPort`)
- Implementation adapter name describes provider (e.g., `OpenAIEmbeddings`, `ChromaVectorStore`)

## Testing Contract
- Contract tests verify adapter fulfills Protocol at runtime
- Use `isinstance(adapter, SomePort)` pattern

## Extensibility
New ports can be added by:
1. Define new Protocol in appropriate `ports/*.py`
2. Document in this AGENTS.md
3. Implement in `infrastructure/adapters/<name>/`
4. Wire in `infrastructure/composition.py`

## Delegation & Boundaries
- Inheritance: Inherits from `src/bhodi_platform/`
- Implemented by `infrastructure/` adapters
- Consumed by `application/` use cases
